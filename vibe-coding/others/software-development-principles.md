# Vibe Coding 必修訓練：在 AI 時代學習寫程式

**作者：Basia Kubicka**

## 10 個軟體開發原則

這些是我們提供給「**天才實習生（Brilliant Intern）**」的基本規則，用來避免 AI 寫程式時最常犯的錯誤。

---

# 1. 分解（Decomposition）

**一次只建構一個模組。**

把軟體開發想像成組裝樂高。

先放上一塊積木，測試沒問題，再放下一塊。

如果整座塔開始搖晃，你立刻知道是哪一塊造成的。

如果一次要求 AI 建立整個 App，它會自行做出數百個假設。

當程式出錯時，你完全不知道問題在哪裡。

因此，應該把工作拆成許多小而獨立的部分，例如：

* 一個頁面
* 一個功能
* 一個函式

逐步完成並驗證。

---

# 2. 不要重複自己（Don't Repeat Yourself，DRY）

**寫一次，到處重複使用。**

AI 為了追求速度，很喜歡把相同的邏輯複製到許多不同檔案。

這是一個陷阱。

因為日後如果需求變更，你可能得修改五個地方，而且很容易漏掉其中一個。

應該把共用邏輯集中寫在一個地方，再讓其他地方引用它。

如此一來：

> 修改一次，就能同步更新所有地方。

---

# 3. 描述性的命名（Descriptive Naming）

**讓任何人都能看懂名稱的用途。**

例如：

如果按鈕是用來儲存食譜，就命名為：

```text
saveRecipe
```

而不是：

```text
button1
```

如果函式負責取得食材資料，就命名為：

```text
fetchIngredients
```

而不是：

```text
getData
```

清楚的命名可以讓：

* 自己
* 團隊成員
* AI

都更容易理解程式碼。

如果 AI 連自己上週寫的程式都看不懂，它就會開始猜測。

而 AI 猜錯的機率非常高。

---

# 4. 清楚的結構與組織（Clear Structure and Organization）

**叉子放叉子區，盤子放盤子區。**

把程式碼想像成廚房。

如果所有東西都丟進同一個抽屜，

沒有人找得到需要的東西。

應該依照用途整理，例如：

* 樣式（Styles）
* 商業邏輯（Logic）
* 頁面（Pages）
* Layout

各自放在固定的位置。

良好的專案結構有兩個好處：

* 出問題時，很容易知道去哪裡找。
* AI 新增功能時，比較知道該把程式放在哪裡，因此犯錯更少。

---

# 5. 嚴格的型別安全（Strict Type Safety）

**資料的拼字檢查（Spell Check）。**

型別安全會檢查：

資料是不是符合預期格式。

例如：

* 是否具有正確欄位？
* 欄位型別是否正確？
* 資料值是否合法？

如果開啟型別檢查，

在程式執行前就能發現錯誤。

如果沒有型別安全：

程式可能平常都能正常運作，

直到某一天：

* API 回傳數字，但你的程式期待的是文字。
* API 少回傳一個欄位。

接著程式突然崩潰，

你卻完全不知道原因。

---

# 6. 關注點分離（Separation of Concerns）

**每一部分只負責一件事情。**

例如：

* Page 負責畫面。
* Service 負責商業邏輯。
* Button 不應直接存取資料庫。

每個模組都應只有一項明確責任。

可以想像郵局：

* 櫃台負責服務客人。
* 後方負責分類包裹。
* 倉庫負責存放包裹。

如果櫃台人員還要同時：

* 分類包裹
* 管理倉庫

整個系統很快就會混亂。

---

# 7. 錯誤處理（Error Handling）

**規劃「事情出錯時」會發生什麼。**

初學者通常只考慮：

> Happy Path（理想流程）

也就是：

一切都正常，

使用者完全照你的預期操作。

但實際情況可能是：

* 網路斷線
* API 無法使用
* 使用者送出空白表單

如果沒有錯誤處理，

App 很可能直接：

* 當機
* 卡住
* 顯示白畫面

有了完善的錯誤處理，

使用者會看到像是：

> 「發生了一點問題，請稍後再試。」

而不是一個壞掉的畫面。

---

# 8. 安全性（Security）

**永遠不要相信 Client。**

**不要把後台的鑰匙交給客戶。**

想像你在郵局工作。

客戶要領包裹時，

你不會直接把倉庫鑰匙交給他。

而是：

1. 驗證身分。
2. 自己去倉庫拿包裹。
3. 再交到櫃台給客戶。

同樣地，

以下這些敏感操作：

* 權限驗證
* 存取資料
* 使用 Secret Key 呼叫 API

都必須在 **後端（Backend）** 完成。

可以把系統比喻成：

* Frontend = 郵局櫃台
* Backend = 倉庫

**使用者永遠不應接觸到後台。**

---

# 9. 無障礙與響應式設計（Accessibility and Responsiveness）

**讓所有人、所有裝置都能正常使用。**

**Accessibility（無障礙）** 表示：

即使使用者：

* 使用螢幕閱讀器
* 使用鍵盤操作

也能順利使用你的 App。

**Responsiveness（響應式）** 表示：

畫面在不同裝置都能良好呈現，例如：

* 手機
* 平板
* 桌機

這些能力：

一開始就做好，

遠比後續補救容易得多。

一個簡單的設計決策，

可能替未來省下數小時的修改成本。

---

# 10. 記錄（Logging）

**留下除錯的麵包屑（Breadcrumbs）。**

每當系統完成重要事件時，

都應該留下紀錄，例如：

* API 呼叫成功
* 登入失敗
* 資料庫錯誤
* 系統例外

這些紀錄就是：

**Log（日誌）**

當系統發生問題時，

Log 就是你追查原因的重要依據。

沒有 Log：

你只能靠猜。

有了 Log：

你可以一步一步追蹤：

* 發生了什麼？
* 哪一步失敗？
* 為什麼失敗？

它就像是：

> **程式碼的監視器（Security Camera）。**

---

# 總結

這 10 個原則其實可以分成三大類：

| 類別                            | 原則                                                                          | 目的                               |
| ----------------------------- | --------------------------------------------------------------------------- | -------------------------------- |
| **可維護性（Maintainability）**     | Decomposition、DRY、Descriptive Naming、Clear Structure、Separation of Concerns | 讓程式容易閱讀、修改、擴充，也讓 AI 更容易理解程式碼。    |
| **可靠性（Reliability）**          | Strict Type Safety、Error Handling、Logging                                   | 降低 Bug、加快除錯速度，提升系統穩定性。           |
| **品質與安全（Quality & Security）** | Security、Accessibility & Responsiveness                                     | 保護使用者資料，並確保所有使用者、各種裝置都能順利使用應用程式。 |

這 10 項原則不只是傳統軟體工程的最佳實踐，在 **AI 輔助開發（Vibe Coding）** 時代更顯得重要。AI 能快速產生程式碼，但若沒有這些原則作為約束，產出的程式往往容易重複、混亂、難以維護，也更容易出現隱藏的錯誤。
