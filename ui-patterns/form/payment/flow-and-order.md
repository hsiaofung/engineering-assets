這堂課的主題是 「流程與順序（Flow and Order）」，重點在於：使用者應該如何一步一步、自然地完成表單。這比單純把欄位排好更重要，因為它會直接影響完成率與使用者信任感。

### 流程與順序（Flow and Order）— 重點整理

### 核心觀念

在正確的時機，詢問正確的資訊。

* 同樣一個問題，問得太早會讓人反感。

* 問得合理、問得有原因，使用者通常願意配合。

* 表單設計不只是 UI 排版，更是 使用者旅程（User Journey）設計。

### 一、以合理順序提問

### 為什麼重要？

使用者會預期流程有邏輯：

* 先建立聯絡方式

* 再提供配送資訊

* 確認內容

* 最後才付款

如果一進頁面就要求信用卡，很多人會直接離開。

### 二、推薦的結帳流程

1

Email

2

手機號碼（選填）

3

寄送地址

4

運送方式

5

檢查訂單資料

6

付款

7

確認頁 / 收據

### 三、付款應該放在後面

### 原則

付款通常是 流程的後段。

### 正確做法

* 先讓使用者完成資料填寫。

* 提供 訂單確認頁（Review Page）。

* 使用者確認金額、商品、地址都正確後，再輸入付款資訊。

### 好處

* 減少付款錯誤

* 降低退款機率

* 增加使用者安心感

### 四、確認頁的角色

確認頁不只是「成功畫面」，它其實像：

電子收據（Receipt）

應包含：

* 訂單編號

* 商品明細

* 金額

* 寄送地址

* 付款方式

* 客服聯絡資訊

這頁也能讓使用者：

* 下載收據

* 列印

* 查看訂單狀態

### 五、說明「為什麼需要這些資料」

這是建立信任的關鍵。

### Email 欄位

### 不要只寫

```
Email
```

### 應該加提示

```
我們會將電子收據寄送到此信箱，並提供退貨、取消或修改訂單的通知。
```

### 手機欄位（選填）

### 好的設計

```
手機號碼（選填）
用於接收運送進度簡訊通知。
```

### 為什麼選填？

* 不是每個人都想收到簡訊

* 不是每個人都有手機

* 尊重使用者選擇權

### 六、Hint / 提示文字的重要性

### 作用

* 降低理解成本

* 減少錯誤

* 增加透明度

* 提升信任感

### 好的 Hint 範例

電子郵件

我們只會用來寄送收據與訂單通知，不會公開你的信箱。

手機號碼（選填）

用於接收物流配送簡訊，可略過。

### 七、只收集真正需要的資料

講師特別強調：

不要用不必要的欄位增加使用者負擔。

### 常見不必要欄位

| 欄位    | 是否必要？ |
| ----- | ----- |
| 公司名稱  | 通常不需要 |
| 生日    | 通常不需要 |
| 性別    | 通常不需要 |
| 備用電話  | 很少需要  |
| 地址第二行 | 可選填   |

### 八、從「資料導向」改成「任務導向」

很多系統會這樣想：

我要收集：姓名、電話、Email、地址、發票…

但 UX 應該這樣想：

使用者現在要完成什麼任務？

### 任務導向流程

| 任務     | 需要的資料  |
| ------ | ------ |
| 建立聯絡方式 | Email  |
| 接收物流通知 | 手機（選填） |
| 配送商品   | 地址     |
| 計算運費   | 運送方式   |
| 完成交易   | 付款資訊   |

這樣設計會自然很多。

### 九、實務上的 Angular / Stepper 對應

你前面學的 Stepper，這裡可以直接套用：

```
Step 1: Contact
  - email
  - phone (optional)

Step 2: Shipping
  - address
  - shipping method
  - note

Step 3: Review
  - order summary
  - edit actions

Step 4: Payment
  - card / Apple Pay / PayPal

Step 5: Confirmation
  - order number
  - receipt
```

### 關鍵細節

* Review 頁一定要能返回修改

* 不要在付款頁才發現地址錯誤

* 每一步都顯示目前進度（1/5、2/5…）

### 十、設計 Checklist（Flow 版）

在設計流程時問自己：

這個欄位現在問，合理嗎？

如果晚一點再問，會不會更自然？

使用者知道我為什麼需要這個資料嗎？

有提供提示文字嗎？

這個欄位能不能改成選填？

有沒有可以刪掉的欄位？

付款前是否有 Review 步驟？

確認頁是否像正式收據？

### 一句話總結

表單設計的重點不是『收集資料』，而是『引導使用者以最少的負擔完成任務』；好的流程會讓使用者覺得一切都發生在剛剛好的時機。

這其實就是 Progressive Disclosure（漸進式揭露） 的應用： 👉 先問最必要的，再根據流程逐步詢問下一步需要的資訊。
